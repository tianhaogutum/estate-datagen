SCENARIO: Protocol dates that do NOT match the contract frequency.

Deliberately introduce AT LEAST ONE of the following anomalies into the
generated data (pick one or combine several):

1. Two maintenance events inside the same interval window.
   Example for QUARTERLY (every 3 months) starting 2024-01-15:
     - Protocol A on 2024-01-20   (normal, k=0)
     - Protocol B on 2024-02-10   (extra, still in Q1)
     - Protocol C on 2024-04-18   (normal, k=1)

2. A completely missing maintenance in one interval.
   Example for QUARTERLY starting 2024-01-15:
     - Protocol on 2024-01-20   (k=0)
     - Protocol on 2024-04-18   (k=1)
     - (2024-07-15 interval has NO protocol — simply skip it)
     - Protocol on 2024-10-12   (k=3)

3. Dates that drift far off the expected cadence (> 30 days away from
   contract_start + k * interval).
   Example for ANNUAL starting 2024-01-15:
     - Protocol on 2024-05-28   (should have been near 2024-01-15)

Requirements:
- The contract's `maintenance_frequency` field must stay consistent and
  realistic (MONTHLY / QUARTERLY / SEMI_ANNUAL / ANNUAL / BIENNIAL).
- Protocol dates must still fall between contract_start and contract_end.
- Keep all other fields coherent (device_type, service_provider, etc.) so
  the only anomaly is the date distribution.
